# Co-simulation runner

This document describes how to build the RTL simulation, build the firmware, and run the cocotb co-simulation flow for the imported Arch2RTL design.

## Prerequisites

Install and verify the following tools:

- Verilator
- Python 3
- cocotb
- pytest
- Cargo / Rust toolchain
- Optional coverage support:
  - verilator_cov-compatible workflow
  - llvm-cov

Verify basic availability:

- `verilator --version`
- `python3 --version`
- `pytest --version`
- `cargo --version`

## Environment overrides

The runner uses the following environment overrides:

- `RTL_TOP=${RTL_TOP:-<RTL_TOP>}`
- `RTL_DIR=${RTL_DIR:-<RTL_DIR>}`
- `FILELIST=${FILELIST:-<FILELIST>}`

If your source workflow provides a build helper in `firmware/validate/verilator_build.md`, prefer that documented flow over ad hoc commands.

## Exact build and run commands

Run from the repository root unless your workflow documents a different working directory.

### 1) Build the RTL simulation

If `firmware/validate/verilator_build.md` exists and contains an explicit build recipe, follow it first.

Otherwise, the default build flow is:

- export the required environment variables
- build the verilator simulation model
- place build outputs under `firmware/validate/`

Example commands:

- `export RTL_TOP=${RTL_TOP:-<RTL_TOP>}`
- `export RTL_DIR=${RTL_DIR:-<RTL_DIR>}`
- `export FILELIST=${FILELIST:-<FILELIST>}`
- `mkdir -p firmware/validate/rtl_build`
- `verilator -Wall --cc --exe --build -Mdir firmware/validate/rtl_build --top-module "$RTL_TOP" -f "$FILELIST"`

If the imported package requires additional include paths or defines, add them through the file list or the generated verilator build instructions.

### 2) Build the firmware

The firmware collateral is Rust-based.

Typical build command:

- `cargo build --manifest-path firmware/Cargo.toml`

If the workspace manifest is located elsewhere in the imported package, use the appropriate Cargo manifest provided by the source workflow.

For validation builds, you may also run:

- `cargo test --manifest-path firmware/Cargo.toml`

Coverage-enabled firmware build example:

- `cargo llvm-cov --manifest-path firmware/Cargo.toml --lcov --output-path firmware/validate/fw_coverage.lcov`

### 3) Run cocotb through pytest

Run the cocotb tests via pytest and direct outputs under `firmware/validate/`:

- `pytest -q firmware/validate --junitxml=firmware/validate/junit.xml`

If your cocotb tests are in a more specific directory or module, point pytest at that location, but keep artifacts under `firmware/validate/`.

The `run_cosim.sh` script provided alongside this document executes a compatible flow using environment overrides.

## Expected outputs

After a successful run, expect the following outputs under `firmware/validate/`:

- RTL build artifacts:
  - `firmware/validate/rtl_build/`
- Test logs:
  - `firmware/validate/cosim.log`
- Optional JUnit XML:
  - `firmware/validate/junit.xml`
- Coverage artifacts, if enabled and supported:
  - `firmware/validate/rtl_coverage.dat` or Verilator-equivalent coverage output
  - `firmware/validate/fw_coverage.lcov`
- Any cocotb-generated waveform or dump files if enabled by the testbench:
  - for example VCD/FST files under `firmware/validate/`

## Notes

- Use the register map and RTL interface generated by the source workflow as the authoritative hardware/software contract.
- The cocotb tests should exercise register reads/writes, reset behavior, and at least one end-to-end transaction path through the imported RTL.
- Keep all generated validation collateral confined to `firmware/validate/` unless the imported package requires a different layout.
