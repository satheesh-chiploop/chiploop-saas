# firmware/integration_contract.md

## Contract Overview
- This document defines the firmware integration contract for the imported Arch2RTL RTL package generated by the source workflow.
- Scope covers register-layer access, driver scaffold behavior, build/cosim collateral, validation hooks, logging, error handling, and software handoff for the actual imported design.
- The contract is intended for Embedded_Run / full-chain integration and assumes the RTL register map and interface definitions are the source of truth.
- Firmware must not invent functionality beyond the generated RTL interface; all APIs map directly to discovered registers, fields, and observed reset/ready behavior.
- System/Host software owns higher-level policy, orchestration, and application use cases; firmware owns hardware control, status exposure, and deterministic error reporting.
- Validation collateral must prove conformance to the imported RTL interface, including reset behavior, register access semantics, and ready/boot status transitions.
- Co-simulation and coverage are enabled and must be used to validate register-layer correctness against RTL.

## Contract Version + Compatibility Policy
- **Contract version:** `1.0.0`
- **Compatibility policy:**
  - **Major version** changes indicate a breaking change in the RTL interface, register map, semantic behavior, or firmware API contract.
  - **Minor version** changes add backward-compatible registers, fields, APIs, or validation hooks.
  - **Patch version** changes fix documentation, validation collateral, or implementation details without changing interfaces.
- Firmware shall expose a version query that reports:
  - contract version
  - RTL package identifier
  - register map revision
  - build timestamp or build identifier
- Forward compatibility is supported only when new RTL fields are reserved/ignored by older firmware and do not alter existing behavior.
- Backward compatibility is supported when firmware continues to honor previously defined reset values, access permissions, and status meanings.

## Interfaces

### Register Layer Interface
The register layer must be generated from the imported RTL register map and used as the only supported hardware access path for the embedded firmware.

| Interface | Type | Expected Behavior | Ownership |
|---|---:|---|---|
| `reg_read(addr, value*)` | API | Reads a 32-bit register at a valid mapped address; returns a defined error for unmapped or access-violating addresses. | FW |
| `reg_write(addr, value)` | API | Writes a 32-bit register at a valid mapped address; honors RO/W1C/W1S/side-effect semantics from the RTL map. | FW |
| `reg_update(addr, mask, value)` | API | Read-modify-write helper for masked fields when supported by the register semantics. | FW |
| `reg_field_get(reg, field)` | API | Returns decoded field value from cached or directly read register data. | FW |
| `reg_field_set(reg, field, value)` | API | Applies field encoding rules defined by the RTL map before writeback. | FW |
| `reg_reset_defaults()` | API | Re-initializes software shadow state to match documented RTL reset defaults. | FW |

### Driver Scaffold Interface
The driver scaffold must be auto-generated or generated-from-spec and remain tightly coupled to the imported RTL package.

| Interface | Type | Expected Behavior | Ownership |
|---|---:|---|---|
| `init()` | API | Verifies device presence, reads version/revision, validates reset defaults, and transitions to ready state if supported. | FW |
| `start()` | API | Applies required runtime configuration and enables operation only after prerequisite checks pass. | FW |
| `stop()` | API | Disables active operation and returns hardware to a safe idle state defined by the RTL behavior. | FW |
| `get_status()` | API | Returns current hardware status in a software-consumable structure mapped from RTL status fields. | FW |
| `get_error()` | API | Returns latched error state and clears only when explicitly specified by RTL semantics. | FW |
| `get_version()` | API | Returns firmware contract version, register map revision, and RTL package identifier. | FW |

### Error Model
Firmware must report only errors that can be justified by the imported RTL interface or integration environment.

| Error Code | Meaning | Recovery |
|---|---|---|
| `FW_OK` | Operation succeeded | None |
| `FW_EINVAL` | Invalid argument or invalid field encoding | Correct caller input |
| `FW_EBUSY` | Hardware not ready or busy | Retry after ready/status check |
| `FW_ETIMEOUT` | Expected hardware transition did not complete in time | Reinitialize or inspect RTL status |
| `FW_EIO` | Register access or device communication failure | Verify bus, reset, or power state |
| `FW_ENOTSUP` | Requested operation not supported by RTL contract | Use supported path only |
| `FW_ESTATE` | Operation not allowed in current device state | Follow state machine sequence |

### Logging Schema
Logs must be stable and machine-parsable for validation and system integration.

| Field | Type | Meaning |
|---|---:|---|
| `ts` | integer | Monotonic timestamp or sequence count |
| `level` | string | `DEBUG`, `INFO`, `WARN`, `ERROR` |
| `module` | string | Firmware module name |
| `event` | string | Stable event identifier |
| `code` | string | Error or status code |
| `detail` | string | Short human-readable context |

Required log events:
- `FW_INIT_BEGIN`
- `FW_INIT_DONE`
- `FW_INIT_FAIL`
- `FW_STATE_CHANGE`
- `FW_REG_ACCESS_FAIL`
- `FW_VERSION_READ`
- `FW_VALIDATE_START`
- `FW_VALIDATE_PASS`
- `FW_VALIDATE_FAIL`

### Power Modes
Firmware must only expose power behaviors that are supported by the imported RTL and platform integration.

| Mode | Expected Behavior | Ownership |
|---|---|---|
| `ACTIVE` | Full normal operation | FW |
| `IDLE` | Quiescent state with clocks/resources preserved as required by RTL | FW/System |
| `RESET` | Hardware/software reset state with documented reset defaults | FW/System |
| `LOW_POWER` | Only if supported by RTL and platform contract; no implicit feature enablement | System/Host |

## Ownership Boundaries

| Domain | FW Owns | System/Host Owns | Validation Owns |
|---|---|---|---|
| Register access | Low-level read/write helpers and field encoding | No direct register semantics assumptions beyond published interface | Confirm access permissions and reset defaults |
| Bring-up | Init/start/stop sequencing | Device discovery and integration policy | Sequence compliance and timing checks |
| Status/error reporting | Stable status/error translation | Application reaction and escalation | Error injection and observability checks |
| Power behavior | State transitions required by RTL contract | Platform power policy and orchestration | State transition validation |
| Logging | Contracted log schema and event codes | Log collection/storage/alerting | Log presence and format checks |
| Versioning | Contract and RTL version exposure | Compatibility decisions in system integration | Version/matrix verification |

## Assumptions
- The imported Arch2RTL workflow produced an RTL register map and interface definition that is authoritative for this contract.
- All supported functionality is representable through mapped registers, reset behavior, and status fields in the source package.
- No undocumented interrupts, DMA paths, or hidden sideband control are assumed in this contract.
- Co-simulation is available against the generated RTL and can be driven by the firmware register API or a compatible test harness.
- Coverage collection is available at least at the register-access and state-transition level.
- Any unsupported RTL feature is treated as `FW_ENOTSUP` rather than inferred by firmware.
- System/Host software will consume only the published APIs, logs, and version fields in this handoff package.

## Validation Hooks
Use the following hooks to test contract compliance against the actual imported design:

| Hook | Method | Pass Criterion |
|---|---|---|
| Register map conformance | Compare generated firmware register definitions to source RTL register map | All addresses, access types, reset values, and field encodings match |
| Reset-default check | Reset device, read all documented registers | Observed values match documented reset defaults |
| Version check | Read version/revision fields at startup | Firmware reports correct RTL package and map revision |
| Read/write semantics | Exercise RO/RW/W1C/W1S/side-effect registers | Hardware behavior matches access type definitions |
| State transition check | Run init/start/stop sequences | Legal transitions succeed; illegal transitions return `FW_ESTATE` |
| Error path check | Force invalid addresses, invalid field values, and busy conditions | Correct error codes and logs are emitted |
| Log schema check | Capture firmware logs during init and failure cases | Required fields and event IDs are present |
| Co-simulation check | Drive register transactions in sim and compare expected vs observed RTL behavior | No mismatches on supported access and status paths |
| Coverage check | Run directed register and state-transition tests | Coverage covers all defined registers, fields, and key transitions |
| Handoff sanity | Validate exported headers, constants, and version strings | Software-facing package is internally consistent |

## Software-Facing Handoff Package
The handoff package for the imported design must include:
- generated register header(s)
- base address and offset definitions
- field masks and shifts
- error code definitions
- version and compatibility macros
- initialization/start/stop/status APIs
- log event ID definitions
- validation test list and expected outcomes
- RTL register map revision identifier
- build identifier for the generated collateral

## Build and Co-Simulation Collateral
- Firmware build shall include the generated register layer and driver scaffold as a standalone module.
- Co-simulation collateral shall provide a register transaction testbench aligned to the imported RTL package.
- Coverage collection shall include:
  - register access coverage
  - reset default coverage
  - status/error coverage
  - legal/illegal state transition coverage
- Validation artifacts must link observed RTL behavior back to the exact generated register definitions used by firmware.

## Validation Report Requirements
The validation report for this contract must include:
- RTL package and register map identifiers
- firmware contract version
- list of validated registers and fields
- pass/fail summary for reset, access, status, and version checks
- any mismatches between generated collateral and RTL source
- coverage summary for enabled coverage points
- co-simulation trace references when enabled

## Handoff Summary
- Firmware is the only layer allowed to interpret the RTL register map at transaction level.
- System/Host software consumes stable APIs, logs, errors, and version fields only.
- Validation owns proof of conformance and must reject any generated collateral that does not match the imported RTL interface exactly.
- This contract is tied to the actual imported design and must be regenerated if the RTL register map or interface changes.
